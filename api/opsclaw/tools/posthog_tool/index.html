PostHog tool. Read-only v1.

Surfaces product-analytics signal that closes the loop between an infra-side alert and what users are actually experiencing:

  • did the error event spike around the time the alert fired?
  • which feature flag was just rolled out?
  • what was this specific user doing when they reported the bug?

Six actions, all GET/POST against the public PostHog API:

  • query_events — count + sample for one event in a time window.
  • recent_flag_changes — flags modified in the last N hours.
  • flag_status — one flag: rollout %, recent metadata.
  • events_for_user — last N events for a distinct_id.
  • session_replay_url — most recent replay URL for a distinct_id.
  • hogql — escape hatch for HogQL/ClickHouse queries.

All actions are read-only; no write surface in v1. See docs/tools/posthog.md for the user-facing summary.

DEFAULT_LIMIT 🔒
Default cap on number of events surfaced per query. Without this, a careless agent can pull millions of rows on a wide HogQL query and spend real money on PostHog egress.
DEFAULT_SINCE 🔒
Default time window when the agent doesn’t specify one. SRE questions are almost always “since the alert fired” which is rarely longer than an hour.
