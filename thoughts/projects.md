# Multi-Project Management

## Problem

Today `TargetConfig` models a single machine or cluster. There is no way to express that a set of targets belongs to the same system — e.g. that `web-1`, `web-2`, `postgres`, and `k8s-prod` are all parts of "acme".

A single OpsClaw agent should be able to manage multiple unrelated systems without targets from one bleeding into the context or runbooks of another.

## Concept

A **project** is a named group of targets that share:

- a context description (what this system is, its SLOs, architecture notes)
- runbooks scoped to that system
- an escalation policy
- an autonomy default that targets inherit if they don't override it

Targets remain the unit of connection (SSH, local, k8s). Projects are the unit of *ownership and context*.

## Config shape

```toml
[[projects]]
name = "acme"
targets = ["web-1", "web-2", "postgres", "k8s-prod"]
autonomy = "approve"           # default for member targets
context_file = "~/.opsclaw/context/acme.md"
escalation = { ... }           # inherited by targets unless overridden

[[targets]]
name = "web-1"
type = "ssh"
# ...
# no autonomy set → inherits from project

[[targets]]
name = "postgres"
type = "ssh"
autonomy = "dry-run"           # overrides project default
# ...
```

A target that belongs to no project continues to work exactly as today. Projects are opt-in.

## Schema changes

Add `ProjectConfig` to `zeroclaw/src/config/schema.rs`:

```rust
pub struct ProjectConfig {
    pub name: String,
    pub targets: Vec<String>,                          // target names
    pub autonomy: Option<OpsClawAutonomy>,             // default for members
    pub context_file: Option<String>,
    pub escalation: Option<serde_json::Value>,
}
```

Add to `Config`:

```rust
pub projects: Option<Vec<ProjectConfig>>,
```

Add validation:
- project names unique
- every target name in a project must exist in `[[targets]]`
- a target may belong to at most one project

## Context layers

Three layers, stacked in this order in the system prompt (most-specific last):

1. **`SOUL.md`** — agent identity and standing instructions ("you are an SRE agent, here is how you behave"). ZeroClaw already loads this; OpsClaw inherits it unchanged.
2. **Project context** (`project.context_file`) — what this system is: architecture, SLOs, team contacts, known quirks.
3. **Target context** (`target.context_file`) — what this specific machine is: its role, what's running on it, host-specific notes.

`SOUL.md` is already generated by the zeroclaw onboard wizard. OpsClaw does not replace it — the SRE-specific instructions live in the project and target context files.

`OpsClawContext` gains:

```rust
pub fn project_for(&self, target_name: &str) -> Option<&ProjectConfig>
pub fn projects(&self) -> &[ProjectConfig]
pub fn targets_for_project(&self, project_name: &str) -> Vec<&TargetConfig>
```

## Autonomy resolution

```
effective_autonomy(target) =
    target.autonomy (if set explicitly)
    ?? project.autonomy (if target belongs to a project)
    ?? OpsClawAutonomy::Approve (global default)
```

This is a pure config-time resolution — no runtime dispatch needed. Resolve it when building `TargetEntry` in `ssh_runner_for` and equivalent methods.

## Daemon

The daemon currently spawns one monitor task per target. Projects add a layer of grouping but do not change the task model — each target still runs its own monitor loop.

What changes: incident memory and runbooks are keyed by `(project, target)` rather than just `target`. This prevents cross-project pattern bleed and allows project-level runbook queries ("what do we know about acme?").

## What does not change

- `TargetConfig` struct — no fields removed or renamed (backward compat)
- Single-target configs continue to work without any projects key
- Audit log format — still per-target-command
- Secret store — project config references target names; no new secret types needed
- The monitor loop and tool dispatch — targets remain the unit of execution

## Setup flow

Project setup is a separate command, not part of the basic onboard:

```
opsclaw onboard     # existing: LLM provider, channels, first target
opsclaw project add # new: group targets into a named project
```

`opsclaw onboard` remains unchanged — it sets up the LLM provider, notification channels, and optionally a first target. It does not prompt about projects.

`opsclaw project add` is run afterwards, once the user has targets configured:

1. Name the project.
2. Select from existing targets (multi-select; only targets not already in a project are shown).
3. Set project-level autonomy default (optional; inherits global default if omitted).
4. Optionally write a context file describing the project.

This keeps onboarding fast and projects opt-in for users who need them.

## CLI

```
opsclaw status                    # all projects + ungrouped targets
opsclaw status --project acme     # targets in acme only
opsclaw project add               # interactive wizard
opsclaw project list              # show all projects and their targets
opsclaw project remove acme       # remove project grouping (targets kept)
```

## Open questions

1. Should the agent maintain one conversation thread per project (shared context across its targets) or one per target? Per-project feels right for correlated incidents ("web-1 and web-2 both started OOMing after the deploy") but needs a multi-target tool dispatch path.

2. Runbook scoping: should runbooks live at project level, target level, or both? Suggest both, with project-level runbooks inherited by all member targets.
